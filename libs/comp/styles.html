About

In this documentation we describe the styles that can be used in a component, which are automatically converted to css classes during the rendering process if the flag __styled__ is set to True.

Introduction

In css, classes are composed of properties. To define a property one needs to set a <value>, which typically depends on some <unit> and can be endowed with further <styles>. So, in css a class can be typically defined as dictionary whose entries are properties. These, in turn, have values given by dictionaries with keys value, unit and styles, as follows:

 1some_class = {
 2    "some_property": {
 3        "value": some_value,
 4        "unit": some_unit,
 5        "styles": [style_1, style_2, ...]
 6    },
 7    "other_property": {
 8        "value": other_value,
 9        "unit": other_unit,
10        "styles": [style_A, style_B, ...]
11    },
12    ...
13
14}

You could try to incorporate all this information into a single string. In other words, one could try to consider strings representing the content of css classes, which could be used in the “class-like” attributes of components: the so-called style strings.

This would allow us to create an inline css framework for comp, in which style strings inside a component, named as above, are automatically converted to underlying css classes, so that their content would be introduced in a <style> tag in the rendered HTML obtained from rendering the given component.

In the following we brief describe how this framework works, whose execution is controlled by the {lib: rendering flag} __styled__.

Property Strings

The vast majority of css classes incorporated in the framework have a single property:

1some_class = {
2    "property": {
3        "value": some_value,
4        "unit": some_unit,
5        "styles": [style_1, style_2, ...]
6    }
7}

Its content can be naturally represented by the following string, which works as a style string for the underlying style property:

<property>-<value><unit>-<style_1>-<style_2>-...

Indeed, suppose we have a component some_comp, defined as follows:

 1from typed import optional, Str
 2from comp import component, Jinja
 3
 4@optional
 5class SomeCompModel:
 6    comp_id: Str,
 7    comp_class: Str,
 8    ...
 9
10@component
11def some_comp(comp: SomeModel, ...) -> Jinja:
12    ...

Then, suppose we created an instance of comp with comp_class containing the style string:

1some_entity = SomeCompModel(
2    comp_id = "some_id",
3    comp_class = "<property>-<value><unit>-<style_1>-<style_2>-...",
4    ...
5)

In this case, render(some_comp, comp=some_entity, __styled__=True) will produce an HTML which begins with the following <style> tag:

<style>
    .[property]-[value][unit]-[style_1]-[style-2]-... {
        [property]: [value][unit] [style_1] [style_2] ...;
    }
    ...
</style>
...

For example, if comp_class = 'margin-top-10px', then the following will be added to the rendered HTML:

<style>
    .margin-top-10px {
        margin-top: 10px;
    }
    ...
</style>
...

Class Strings

A few css classes with multiples properties are available as style strings. Instead of using the same strategy above to build them, which would become highly understandable with the increasing of the number of properties, for that cases we select a word to represent the entire class:

<word>-<value_1><unit_1>-<value_2><unit_2>-...

For example, if in some_entity of SomeCompModel we set comp_class = 'left-10px', representing a left alignment with a padding of 10px, then the following will be attached to the rendered HTML:

<style>
    .left-10px {
        display: flex;
        align-items: center;
        justify-content: flex-start;
        padding-left: 10px;
    }
</style>

Aliases

To each style string representing a css class there corresponds a number of other style strings that represents the same css class: the so-called style aliases. The idea is to cover most of the user expected behavior.

For the case of style strings associated with some style property, the alias is of the property. On the other hand, for style strings representing classes with multiple properties, the alias is of the word describing the class.

For instance:

  1. instead of margin-top-10px one could use mt-10px

  2. instead of left-10px one could use lft-10px or just l-10px, or even left-center-10px, lft-cnt-10px, l-c-10px or lc-10px.

Prefixes

One time constructed a style string, one can append it with a {lib:style prefix}, producing a new style string which works only inside a specific {lib:style scope}.

Notation

In the remaining part of this document we present the acceptable style strings, their underlying style property (if any), their meaning, their style aliases and the available {lib:style prefixes}.

In order to do that we will use the following notations:

Notation.

  1. <x> will represent a possible value in the values column

  2. <u> will represent a possible value in the unities column

  3. <s> will represent a possible value in the styles column

Also, the variables <x>, <u> and <s> will be omitted in the aliases column.

  1. flex-left-center-<x><u> will apply a left padding of <x><u>

  4. and so on for each alias.
