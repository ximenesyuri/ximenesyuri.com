About

In this documentation we describe the styles that can be used in a component, which are automatically converted to css classes during the rendering process if the flag __styled__ is set to True.

Introduction

In css, classes are composed of properties. To define a property one needs to set a <value>, which typically depends on some <unit> and can be endowed with further <styles>. So, in css a class can be typically defined as dictionary whose entries are properties. These, in turn, have values given by dictionaries with keys value, unit and styles, as follows:

 1some_class = {
 2    "some_property": {
 3        "value": some_value,
 4        "unit": some_unit,
 5        "styles": [style_1, style_2, ...]
 6    },
 7    "other_property": {
 8        "value": other_value,
 9        "unit": other_unit,
10        "styles": [style_A, style_B, ...]
11    },
12    ...
13
14}

You could try to incorporate all this information into a single string. In other words, one could try to consider strings representing the content of css classes, which could be used in the “class-like” attributes of components: the so-called {lib:style strings}.

This would allow us to create an inline css framework for comp, in which {lib:style strings} inside a component, named as above, are automatically converted to underlying css classes, so that their content would be introduced in a <style> tag in the rendered HTML obtained from rendering the given component.

In the following we brief describe how this framework works, whose execution is controlled by the {lib: rendering flag} __styled__.

Single Properties

Let us first consider the case of a class with a single property:

1some_class = {
2    "property": {
3        "value": some_value,
4        "unit": some_unit,
5        "styles": [style_1, style_2, ...]
6    }
7}

Its content can be naturally represented by the following string, which will work as a {lib:style string} for the underlying style property:

<property>-<value><unit>-<style_1>-<style_2>-...

Indeed, suppose we have a component some_comp, defined as follows:

 1from typed import optional, Str
 2from comp import component, Jinja
 3
 4@optional
 5class SomeModel:
 6    comp_id: Str,
 7    comp_class: Str,
 8    ...
 9
10@component
11def some_comp(comp: SomeModel, ...) -> Jinja:
12    ...

Then, suppose we created an instance of comp with comp_class containing the {lib:style string}:

1some_entity = SomeModel(
2    comp_id = "some_id",
3    comp_class = "<property>-<value><unit>-<style_1>-<style_2>-...",
4    ...
5)

In this case, render(some_comp, comp=some_entity, __styled__=True) will produce an HTML which begins with the following <style> tag:

<style>
    [property]-[value][unit]-[style_1]-[style-2]-... {
        [property]: [value][unit] [style_1] [style_2] ...;
    }
    ...
</style>
...

For example, if comp_class = 'margin-top-10px', then the following will be added to the rendered HTML:

<style>
    margin-top-10px {
        margin-top: 10px;
    }
    ...
</style>
...

Multiple Properties

For the case of classes with multiple properties, one could try to use the same structure. Indeed, for a class like

 1some_class = {
 2    "property_1": {
 3        "value": value_1,
 4        "unit": unit_1,
 5        "styles": [style_1_1, style_1_2, ...]
 6    },
 7    "property_2": {
 8        "value": value_2,
 9        "unit": unit_2,
10        "styles": [style_2_1, style_2_2, ...]
11    },
12    ...
13}

one could think in using:

(<property_1>-<value_1><unit_1>-<style_1_1>-<style_1_2>)-(<property_2>-<value_2><unit_2>-<style_2_1>-<style_2_2>)-...

However, notice that this becomes highly understandable when the number of property grows. At the same time, it is completely flexible, in the sense that you can control any parameter of any property of the class directly in the representing string.

The majority of the classes described by {lib:style strings} will correspond to a single property, so that the first case will apply.
